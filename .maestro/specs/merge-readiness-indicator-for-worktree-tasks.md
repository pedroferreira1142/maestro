# Merge-readiness indicator for worktree tasks

Surface a live merge-readiness badge on every worktree task entry in the session sidebar, so the outcome of the Merge button is known before clicking it. Extend WorktreeTaskState in src/shared/types.ts with conflict-prediction fields (e.g. conflictFiles: string[] | null) and have SessionManager.getWorktreeTaskState (src/main/SessionManager.ts) populate them via the existing GitService.mergeConflictFiles, aheadCount, and dirtyCount helpers (src/main/GitService.ts) — all read-only operations that never touch the base repo's working tree, index, or HEAD. The renderer (src/renderer/src/store.ts) polls the existing worktree:state IPC channel (src/main/ipc.ts) periodically for visible worktree sessions and caches results per session; SessionSidebar.tsx renders the badge next to the existing 'branch → baseBranch' line, alongside the Merge button. States: 'no commits' (nothing ahead of base), 'clean' (commits ahead, merge-tree predicts no conflicts), 'conflicts (N)' (N conflicted files), 'uncommitted' modifier when the worktree is dirty, and a muted 'unknown' when prediction is unavailable (missing refs, old git, deleted folder).

## Specs

- [ ] Every worktree task entry in the session sidebar shows a compact merge-readiness badge in exactly one of these states: 'no commits' when the task branch has no commits beyond the base branch, 'clean' when commits exist and a git merge-tree dry-run predicts no conflicts, 'conflicts (N)' with the conflicted-file count when it predicts conflicts, and a muted 'unknown' state when the prediction cannot be computed (e.g. branch deleted, worktree folder missing, or merge-tree unsupported).
- [ ] Hovering the badge shows a tooltip: for the conflicts state it lists the conflicted file paths (capped at 10, with '+N more' beyond that); for the clean state it shows the number of commits ahead of the base branch; for all states it notes the number of uncommitted files in the worktree when that count is greater than zero.
- [ ] When the worktree has uncommitted changes, the badge carries a visible 'dirty' indicator (e.g. a dot or asterisk) in addition to its main state, so the user knows a commit-first merge would include extra work.
- [ ] Computing the badge never mutates any repository: the base repo's working tree, index, HEAD, and the worktree itself are byte-identical before and after a readiness check (only read-only commands such as merge-tree --write-tree, rev-list --count, and status are used).
- [ ] Badges refresh automatically without user action: each visible worktree session's state is re-checked on a periodic interval (between 15 and 60 seconds) and shortly after that session's status transitions from 'working' to 'idle' or 'needs-attention', so the badge reflects commits Claude just made.
- [ ] Clicking the badge forces an immediate re-check of that task's merge readiness, and the badge shows a transient pending/checking appearance until the fresh result arrives.
- [ ] Non-worktree sessions show no merge-readiness badge anywhere, and the existing Merge / Remove buttons and their confirmation flows continue to behave exactly as before.
- [ ] A readiness check that fails (git error, missing folder, deleted branch) degrades to the 'unknown' badge state silently — it never raises a dialog, OS notification, or console error spam, and never blocks the merge button from being used.
