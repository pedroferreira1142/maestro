# Conversation History Recall — full-text search across all past Claude conversations

Add a search dialog that indexes the text of every past Claude Code conversation on disk and returns matching snippets, so a developer can find the conversation where a problem was previously solved and continue it. Main side: add `searchConversations(query)` to `src/main/ConversationsService.ts` that walks every `~/.claude/projects/*/*.jsonl` (the same directory UsageService scans), extracts message text with the existing `messageText` helper, and returns hits — reusing a per-file mtime/size cache like `UsageService.entriesFor` so repeated searches stay cheap. Each hit carries the conversation id, the conversation's real working directory (read from the transcript line's `cwd` field — present on user/assistant lines, e.g. `C:\repos\session-manager-ai`), `lastActivityAt`, total match count, and a text snippet around the first match. Wire it through `src/main/ipc.ts` (`conversations:search`), `src/preload/index.ts`, and `src/shared/api.ts` + a new `ConversationSearchHit` type in `src/shared/types.ts`. Renderer: a new `ConversationRecallDialog.tsx` modeled on `GlobalSearchDialog.tsx` (debounced input, grouped/snippet rows, keyboard nav), opened via a keyboard shortcut wired in `App.tsx` and a `CommandPalette.tsx` entry, gated by a `historyRecallOpen` flag + open/close actions in `src/renderer/src/store.ts`. Continuation reuses existing flows: if a hit's `cwd` matches an open session's folder, focus it and open the existing `ResumePickerDialog`; otherwise create a new session in that folder resuming the conversation via `confirmNewSession`'s `resumeConversationId` path.

## Specs

- [ ] Calling `window.api.searchConversations(q)` with a query of at least the minimum length returns only conversations whose extracted user/assistant text contains `q` (case-insensitive), and returns an empty array (never throwing) for a blank or too-short query or when `~/.claude/projects` is missing.
- [ ] Each returned hit includes the conversation id, the real repo working directory taken from the transcript's `cwd` field, the conversation's last-activity timestamp, the total number of matches in that conversation, and a snippet of surrounding text containing the first match.
- [ ] Opening the recall dialog via its keyboard shortcut (and via its Command Palette entry) shows an empty/hint state until at least the minimum number of characters is typed, then displays result rows; the dialog opens and closes independently of every other dialog.
- [ ] Each result row renders the snippet with the matched query substring visually highlighted, labels the row with the repo (folder basename) and a relative "time ago" of last activity, and rows are ordered most-recently-active first.
- [ ] Search is debounced and bounded: total results and per-conversation matches are capped so a very common term does not freeze the UI, and an explicit "no matches" message is shown when a valid query finds nothing.
- [ ] Choosing a result whose `cwd` equals an already-open session's folder focuses that session and opens the existing per-session Resume picker for that conversation, so the conversation is only resumed after the user confirms (no automatic terminal restart on click).
- [ ] Choosing a result whose `cwd` has no open session starts the new-session flow for that folder with the conversation pre-selected to resume, producing a claude terminal launched with `--resume <conversationId>`.
- [ ] Running a search performs no writes: no transcript file, session, or setting is modified until the user explicitly continues a conversation, and the pre-existing Global Search (live scrollback) and per-session Resume picker continue to behave exactly as before.
