# Resume picker — pick which past conversation to continue

Today PtySession can only resume a session's most recent conversation via a blind `--continue` (StartMode is just 'fresh' | 'continue'; PtySession.spawn appends `--continue` and otherwise nothing). Add an optional picker that lists the prior conversations for a session's repo folder and resumes a chosen one with `--resume <id>`. Reuse the transcript-reading approach in src/main/UsageService.ts: conversations live at `~/.claude/projects/<encoded>/*.jsonl`, where the encoding matches Claude Code's (every non-alphanumeric char replaced by '-', see encodeFolder in StatusBar.tsx/UsageWidget.tsx), the conversation id is the .jsonl filename without extension, and each line is a JSON object with a `type` ('user'/'assistant'), `message`, and `timestamp`. Add a new main-process lister (e.g. a ConversationsService or a function beside UsageService) exposed through src/shared/api.ts + src/preload/index.ts + src/main/ipc.ts. Surface the picker (a) at new-session creation in NewSessionDialog.tsx and (b) from a session context-menu entry 'Resume a different conversation' in SessionSidebar.tsx. To spawn the chosen conversation, set the claude terminal's claudeArgs to include `--resume <id>` and use startMode 'fresh' so PtySession does NOT also append `--continue` (PtySession already spreads config.claudeArgs into the spawn args). Do not change StartMode semantics or the existing default paths.

## Specs

- [ ] A new main-process API (reachable from the renderer via window.api) returns the list of prior conversations for a given repo folder by reading `~/.claude/projects/<encoded>/*.jsonl` using the same folder encoding as Claude Code (every char not matching [a-zA-Z0-9] replaced with '-'); each item contains the conversation id (the filename without the .jsonl extension), a last-activity timestamp, a count of messages, and a one-line preview of the first user message, sorted most-recent-first.
- [ ] When the encoded project directory does not exist or contains no .jsonl files, the API returns an empty array (no throw), and the picker UI shows an explicit empty state (e.g. 'No past conversations for this repo') rather than an error.
- [ ] Conversation listing skips malformed or partial JSONL lines instead of failing, mirroring UsageService.parseFile's tolerant per-line try/parse, so a half-written transcript still lists with the entries that did parse.
- [ ] Selecting a conversation in the picker spawns the claude terminal with the CLI flag pair `--resume <id>` (the chosen conversation's id) and that conversation's history appears in the terminal — verifiable by asserting the resolved spawn args contain '--resume' followed by the selected id.
- [ ] A claude terminal launched via the resume picker is invoked with exactly one resume flag: `--resume <id>` is present and `--continue` is absent in the same invocation.
- [ ] The picker is reachable both from the new-session creation flow (NewSessionDialog) and from a per-session context-menu entry labelled 'Resume a different conversation' (SessionSidebar); the new-session flow keeps a default option that starts a fresh session with no resume flag.
- [ ] When the user cancels the picker or does not choose a conversation, behaviour is unchanged: new sessions start fresh and the pre-existing `--continue` resume action still resumes the most recent conversation exactly as before.
- [ ] The first-user-message preview is derived from the conversation's first entry with type 'user', using its textual message content (ignoring tool-result/meta-only entries) and truncated to a single line for display.
